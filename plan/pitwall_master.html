02API Key & Pricing
OpenF1 live data costs $10/month per user for real-time access. Pitwall does not proxy this cost — users connect their own OpenF1 API key. Historical data (2023+) is free with no key required. This is disclosed clearly at first launch.
No workaround exists for live F1 data. Real-time OpenF1 requires a paid subscription. Pitwall cannot aggregate or relay this on users' behalf without violating OpenF1's terms. The honest approach: require users to provide their own key, and make the value proposition clear enough that $10/month is obvious value.

Key storage & security
The OpenF1 key is stored in the browser's localStorage (desktop app: OS keychain). It is never sent to Pitwall's servers. All OpenF1 API calls are made directly from the user's browser to api.openf1.org. No key = no liability for Pitwall.
Key validation flow

App fires /sessions?session_key=latest as a validation probe
200 response → key valid, store and proceed to live mode
401/403 → show "Invalid key" error with link to OpenF1 billing
429 → show rate limit warning, suggest waiting 60s
03Ambient Race Layer
The app environment itself communicates race state. The banner shifts color with every flag change. A brief animated toast announces what changed — then fades, leaving the gradient as a passive signal. RGB smart lights follow the same events.
Green
rgb(0,200,80)

Safety car
rgb(255,165,0)
Fastest lap
rgb(155,89,245)
Toast behavior: On any flag/state change, a brief pill-shaped toast slides in over the ambient bar (300ms ease-out), holds for 2 seconds showing the race control message, then fades out (600ms ease-in). After it fades, the gradient color carries the meaning passively — no text remains. This means glancing at the screen while watching TV communicates race state through color alone.
Leader color mode
During green flag, the ambient bar tints to P1 constructor's color. On a lead overtake, the new team color flashes for 800ms then settles over 1.2s. Togglable off. RGB lights follow identically.
Calm mode
Strips all color to warm white rgb(237,232,220). Flag changes still show as text-only toasts. Useful for concentrated data work or users who find the color theatrical. RGB output → warm white or off.
04Window & Workspace System
Workspaces are named configurations. Windows are physical browser windows showing one canvas from any workspace. Multiple windows in the same workspace sync live via BroadcastChannel API. Windows in different workspaces are fully independent.
Example multi-screen configurations
SetupWorkspacesSync behavior
2 monitors, 1 teamBoth in "Race day"Changing focus driver in either window updates both. Different canvas tabs can show different content.
2 monitors, 2 stories"Race day" + "NOR focus"Fully independent. VER on monitor 1, NOR on monitor 2. No cross-workspace contamination.
3 monitors2× "Race day" + 1× "Strategy"The two Race day windows sync with each other. Strategy window is independent.
Technical sync mechanism: BroadcastChannel API handles same-workspace sync with ~16ms latency. For cross-device sync (TV + laptop), a lightweight WebSocket room keyed to the workspace ID propagates state changes within the same session. No server-side persistence — purely in-session relay.
05Driver Management
Drivers have team colors, numbers, and positions. The focus system has three tiers — canvas inherit (default), position role (P1/P2/P3/GAP+1/GAP-1), and specific named driver. The compact focus bar shows only your starred drivers.
Canvas inherit
Default for all widgets. Follows whatever driver the canvas is focused on. Switching canvas focus updates all inherit widgets instantly.
Position role (P1/P2/P3…)
Widget auto-follows the live race position. P1 widget silently switches to whoever takes the lead. GAP±1 is relative to canvas focus driver.
Specific driver pin
Widget permanently locked to one named driver regardless of position or canvas focus changes. Shown with purple dashed border.
Team color data: Each driver is mapped to their constructor's official hex color via a season asset file bundled with the app. Colors update on new season load. Used across: focus bar, position map dots, ambient bar leader mode, radio channel indicators, car visualization. Team logos stored as inline SVG assets.

07Car Visualization Widget
A 2026-regulation-accurate SVG side profile of the F1 car. Team colors applied as a livery stripe — no full livery replication to avoid IP issues. Tyre compound/wear/age badges float directly on top of the tyre circles. Damage and telemetry callouts are integrated into the body.
2026 regulation profile elements
✓Long ground-skimming nose with S-duct
✓Large rounded sidepods with aggressive undercut channel
✓Halo flush with cockpit surround, center pillar
✓Engine cover hump → roll hoop / airbox
✓Short, wide rear wing with DRS gap indicator
✓18" wheels with 5-spoke rims and Pirelli compound rings
✓Rear tyre wider than front (accurate sizing ratio)
Tire data overlay
Each tyre has a compound badge (SOFT/MED/HARD + color), a lap age, and a wear progress bar — all positioned directly on top of the tyre SVG circle. FL/FR float above and below the front tyre. RL/RR float above and below the rear. Wear bar color: green → amber at 28% → red at 50%.

09Advanced Radio Scanner
Scanner-style multi-channel radio with four listening modes, per-driver channel toggles, squelch, key-up SFX (synthesized via Web Audio API), auto-transcription via Whisper, and keyword alert badging.
Key-up SFX detail
80ms synthesized radio click plays before each clip. Generated client-side via Web Audio API — no file to load, zero latency. Volume adjustable separately from clip volume. Classic push-to-talk feel.
Squelch
Clips below amplitude threshold are silenced entirely — no dead air. Threshold adjustable 0–100%. Default 30%. Uses Web Audio AnalyserNode to measure peak amplitude before allowing playback.
Keyword alerts
Whisper transcription runs on each clip. Clips containing "box", "cliff", "unsafe", "retire", "push" get a color badge and pin to top of feed. User-configurable keyword list. Searchable session log.
10F1TV & Stream Overlay
Phase 4 / long-term feature. Pitwall runs as a transparent overlay on top of an F1TV (or any stream) window. The app does not embed or restream F1TV content — it overlays external windows with data widgets. This requires a desktop app (Electron or Tauri).
Scope boundary: Pitwall never captures, records, or restreams F1TV content. The overlay sits above the browser/app window at the OS compositor level — it draws on top, reads nothing from the stream. F1TV terms of service compliance is the user's responsibility. Pitwall's role is purely additive data display.

Log access
The diagnostic log is accessible at any time from Settings → Diagnostics. Logs are stored in memory (last 1000 entries) and optionally written to localStorage. Export as JSON for reporting issues. A "Report bug" button packages the last 100 log entries with app version and sends to a configured endpoint (Sentry/PostHog).
Error telemetry (optional)
With user consent, unhandled errors are forwarded to a Sentry project. No personal data, no usage tracking — only stack traces and error codes. Consent is opt-in during onboarding. Users can revoke at any time from Settings → Privacy. All error codes follow the ERR_* namespace shown in screens above.
12Inference Engine
Three data tiers: raw API fields, derived calculations, and inferred metrics built on user-editable formulas. All inferred values are tagged ~EST to communicate uncertainty. The formula builder lets advanced users and the community improve models over time.
Inference accuracy reference
MetricSource fieldsFormula basisAccuracy
Tyre cliff lap/stints /laps /weatherBase compound window ± track temp deviation ± observed deg rate ± SC credit±3 laps

Pit urgency scoreDerived from above(laps_to_cliff/5) + (1−gap_behind/3) + undercut_closing_bonus · clamped 0–1~70%
Fuel level/car_dataModeled consumption rate from throttle integral · ±10kg uncertainty at race end~45%
Formula builder: Every inferred metric's formula is editable via Widget Settings → Formula tab. The editor shows the current expression with syntax highlighting, lets you reference any raw API field by name, and previews results against historical session data in real time. Community formulas are published to a shared registry — load any formula with one click, fork it, improve it, share the update.
