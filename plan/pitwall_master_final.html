01North Star & Data Layer

Pitwall is an F1 race intelligence platform built on the OpenF1 API. The goal is to give fans access to the same quality of real-time data that professional teams work with, in an interface that is fully configurable, shareable, and that responds to the environment around the user.

Every widget is drag-and-drop. Every inferred metric is built on a user-editable formula. Layouts export as compact code strings that can be shared anywhere without an account. Smart lights and the app environment respond to flag state and race events. The app is not affiliated with Formula 1, FIA, or Formula One Management.

Core principles
PrincipleDefinition
No lock-inLayouts export as plaintext codes. No account needed to share. The app works meaningfully without a paid API key (historical mode).
HonestyAll inferred metrics are labeled ~EST. Accuracy figures are documented. The app never presents estimates as official data.
ConfigurableEvery widget has settings. Every inferred formula is editable. Every layout element is moveable and resizable.
CommunityLayouts, formulas, and widget configurations can all be shared. The community improves the inference engine over time via a public formula registry.
PeripheralThe ambient layer is designed for peripheral perception. A user watching the race on TV should absorb race state through color alone without looking directly at the app.
02API Key & Pricing

OpenF1 live data requires a paid subscription at $10/month per user. Pitwall does not proxy or relay this feed. Users provide their own OpenF1 API key to enable live mode. Historical data from 2023 onward is free and requires no key. This is disclosed clearly at first launch with two explicit tiers.

No live workaround exists. Real-time OpenF1 data requires a paid user subscription. Pitwall cannot aggregate or relay live data on behalf of users without violating OpenF1 terms of service. The key is stored locally and never sent to Pitwall servers. All API calls originate directly from the user's browser to api.openf1.org.

03Ambient Race Layer

The ambient layer is a persistent banner at the top of every canvas view that communicates race state through color. When race state changes, a brief toast notification appears over the banner, shows the relevant race control message, then fades out. After the toast fades, the gradient color continues to carry the meaning passively. Users watching the race on TV should absorb race state through peripheral vision without looking at the app.

Design principle: The toast announces. The gradient remembers. After the text fades, a fan glancing up absorbs race state through color alone — no reading required.

06Canvas & Layout System

A canvas is a named drag-and-drop widget grid within a workspace. Each canvas autosaves to localStorage every 3 seconds. Canvases can be exported as a compact layout code string encoding all widget positions, sizes, driver contexts, formula overrides, and ambient settings. The code can be shared anywhere and imported on any device — no account needed on either end.

Workspace and canvas instructions
Click + to the right of the tab row to create a new canvas. Give it a name.
Each canvas has its own independent layout, focus driver, and settings. Switching tabs never affects other canvases.
Double-click any tab label to rename. Drag tabs to reorder. Right-click → Close canvas to remove (prompts to confirm if unsaved).
Right-click any tab → Open in new window to pop the canvas out to another monitor. Both windows remain in sync via BroadcastChannel.
Canvases autosave every 3 seconds. Green dot in toolbar = saved. Spinner = saving in progress.
Widget management instructions
Click + Widget in the toolbar to open the widget palette. All available widget types are listed by category with a one-line description and data source tag.
Click a widget type to place it at 1×1 on the canvas, or drag it directly to a grid position. All new widgets default to FOCUS driver context.
Toggle Edit mode to enable drag and resize. Widgets are locked outside Edit mode to prevent accidental changes during a live race.
Drag any widget to a new grid position. Snap is automatic. Drag the bottom-right handle to resize. Minimum and maximum sizes are enforced per widget type.
Right-click any widget to open its settings panel. Configure driver source, display options, and formula overrides. Changes apply immediately.
Right-click → Remove, or click the × handle in Edit mode, to remove a widget.
Layout export and sharing
Click Export in the toolbar. The layout code is copied to clipboard and shown in a panel below.
The layout code is a compact human-readable string encoding: widget positions, sizes, driver contexts, ambient toggles, radio config, and formula overrides. Contains no personal data.
To import: paste the code into the import field (workspace menu or + tab menu). Canvas loads immediately for preview before confirmation.
Download a .pitwall file for a full JSON representation including all formula overrides. More complete than the code string. Useful for personal backups.
Share the layout code anywhere — Reddit, Discord, X, group chats. No account needed on either end. The code works on any device with Pitwall open.
Layout code example — compact format
pw:v1 · ws:"Race day" · focus:VER · theme:dark
widgets:[lap_delta·2x1, ers_deploy·1x1, tyre_age·1x1@NOR, traffic·1x1, battery·1x1, weather·1x1]
ambient:on · leader_color:on · rgb:wled · radio:scanner

Tyres & Strategy
6 widgets
Y.1
Tyre Intelligence
Inferred
min 1×1
Compound, lap age, lap-time degradation slope (expressed as seconds per lap lost), predicted cliff lap, and pit urgency score. The cliff prediction formula is user-editable. Warning badges activate at configurable thresholds: amber at N laps before predicted cliff (default 5 laps), red at 2 laps. The pit urgency score is a composite 0–1 metric described under Y.4. All values labeled ~EST.
Data source
/stints — compound, tyre_age · /laps — lap time trend for deg rate · /weather — track_temp for cliff model
Deg rate calc
Linear regression on the last 5 non-traffic, non-SC laps of the current stint. Outliers beyond 2σ are excluded. Expressed as seconds per lap.
Cliff formula
Default: BASE_WINDOW[compound] - (track_temp - 45) × 0.3 - deg_rate × 1.8 + sc_laps × 2.1. BASE_WINDOW: SOFT=18, MEDIUM=28, HARD=40, INTER=35, WET=50. Editable in Formula tab.
Accuracy
±3 laps under typical conditions. Degrades in non-standard track temperatures or when SC laps skew the deg rate calculation.

Squelch

Clips are analyzed for peak amplitude before playback using a Web Audio AnalyserNode against a brief pre-buffer. Clips where peak amplitude falls below the squelch threshold are silenced entirely — the feed row still appears with a muted indicator icon, but no audio plays. The threshold is adjustable from 0% (no squelch) to 100% (nothing plays). Default is 30%. The recommended range for eliminating dead air while retaining all speech is 20–40%.

Key-up sound effect

An 80-millisecond synthesized radio click plays before each audio clip. The sound is generated client-side using the Web Audio API — no audio file to load. The synthesis uses a short burst of bandlimited white noise through a fast attack/decay envelope, producing a sound characteristic of a push-to-talk radio activating. Key-up SFX volume is controlled by a separate slider from clip volume. Default 60%. Can be disabled entirely in widget settings.

Auto-transcription pipeline

Each audio clip URL is passed to a Whisper endpoint immediately on receipt. Transcription typically completes within 1–2 seconds. The endpoint can be self-hosted (recommended for privacy) or a cloud API endpoint configured in Settings → Integrations. Transcripts are stored against their clip timestamp and displayed in the feed row below the play button. All transcripts in the session are searchable and exportable as plain text or JSON.

Keyword alerts

After transcription, each clip is scanned for a configurable keyword list. When a match is found, the clip row receives a colored badge identifying the matched keyword and is elevated to the top of the feed regardless of the active listening mode. Default keywords: box, cliff, unsafe, retire, push, tyre, safety car, rain. The list is editable in widget settings and persists in the layout JSON.

14Inference Engine

OpenF1 does not provide ERS state, battery level, tyre wear percentage, fuel load, or engine mode — these are proprietary team telemetry values not exposed by the API. Pitwall infers approximations of these metrics from the available raw data. All inferred values carry an ~EST label and documented accuracy figures. The formulas are user-editable and the community can publish improvements to a shared formula registry.

15OpenF1 API Reference

All data is sourced from the OpenF1 REST API at https://api.openf1.org/v1/. Historical data (2023 onward) is freely accessible without authentication. Live data requires a Bearer token from the user's OpenF1 subscriber credentials. Rate limits: 6 req/s and 60 req/min for subscribers; 3 req/s and 30 req/min for anonymous access.
